# Architecture – Min Hverdag

The guiding principle is **straightforward code another coding agent can easily
understand and extend**. No plugin framework, workflow engine, event bus, or
dependency-injection layer — just modules organised by domain.

## Module boundaries

```
src/
  main.tsx                 App entry; mounts <App/>, imports global styles
  App.tsx                  Providers, routing, hydration + one-time seeding

  store/                   The single source of truth
    types.ts               All entity types + the AppData document shape
    defaults.ts            CURRENT_VERSION + emptyAppData()
    store.ts               Zustand store: state + actions + persistence
    selectors.ts           Pure, derived read helpers

  persistence/             Everything about storing & moving data
    idbStorage.ts          IndexedDB adapter (zustand StateStorage)
    migrations.ts          Versioned migration framework
    backup.ts              export / validate / import of JSON backups

  data/
    lifeAreas.ts           Fixed life-area definitions
    seed.ts                Realistic seed document

  components/              Shared, presentational UI
    TopBar, BottomNav, CheckRow, ToastProvider

  features/                One folder per domain
    today/                 TodayScreen
    tasks/                 TasksScreen, TaskForm, PriorityTag
    tracks/                Tracks + detail screens, TrackForm, IdeaComposer,
                           logic.ts (focus rule), areaVisuals.ts
    routines/              Routines + detail screens, logic.ts
    workouts/              Workouts + detail + player, logic.ts
    settings/              SettingsScreen (backup/import/reset)

  lib/                     Tiny cross-cutting helpers (dates, ids)
  styles/global.css        Theme tokens + component styles
  test/setup.ts            Vitest setup (fake-indexeddb, jest-dom)
```

**Domain logic lives in `features/*/logic.ts` as pure functions** (e.g. the
max-two-focus rule, today's-workout resolution, routine completion). This keeps
the important rules unit-testable without React and reusable from both the store
and the UI.

## State

- A single **Zustand** store holds the whole `AppData` document plus action
  functions. There is one document, not many slices — this matches the
  local-first, single-user product and makes export/import trivial.
- Components read via `useStore(...)` and mutate via the store's action methods.
- Store actions delegate to pure logic helpers where a real rule exists
  (`setTrackFocus`, `toggleRoutineStep`), so the rule has one implementation.
- Derived reads (focus tracks, important tasks, next action, task/action
  separation) are pure functions in `store/selectors.ts`.

## Persistence

- The store uses Zustand's `persist` middleware with a **custom IndexedDB
  storage adapter** (`persistence/idbStorage.ts`, built on `idb`). The entire
  document is serialised under one key (`app-data`) in one object store.
- **Hydration**: `persist` rehydrates asynchronously on load. `App` waits for
  `_hasHydrated` before rendering, then calls `seedIfEmpty()` — seed data is
  installed **only** when `seeded` is false (i.e. no user data yet).
- **Migrations**: `persist` is configured with `version: CURRENT_VERSION` and a
  `migrate` that routes through `persistence/migrations.ts`. Migrations are an
  ordered map `N → N+1`; a document of any version is upgraded one step at a
  time. Version 0 = any legacy/unversioned document. See DATA_MODEL.md.
- **Backup**: `persistence/backup.ts` exports the document as pretty JSON and
  validates untrusted imports (JSON parse, object shape, numeric version, not
  from a newer version, correct field types) before migrating and returning it.
  The UI shows a replace warning before applying an import.

Why a single-document store? It keeps persistence, export/import and migrations
simple and consistent, and the data volume for one person is tiny. If the data
ever outgrows this, the storage adapter is the seam to split into multiple keys.

## Routing

- `react-router-dom` with **HashRouter**. Hash routing avoids 404s on refresh
  and deep links on GitHub Pages (no server rewrites needed), which also suits an
  installed PWA.
- Routes: `/today`, `/tasks`, `/tracks`, `/tracks/:trackId`, `/routines`,
  `/routines/:routineId`, `/workouts`, `/workouts/:workoutId`,
  `/workouts/:workoutId/play`, `/settings`. Unknown paths redirect to `/today`.
- Persistent bottom navigation is rendered by `App` outside `<Routes>`.

## PWA

- `vite-plugin-pwa` (`registerType: 'autoUpdate'`, Workbox `generateSW`) creates
  the service worker and precache. A web manifest declares name, colours,
  `display: standalone`, `start_url`/`scope` = the base path, and icons.
- Icons in `public/` (192, 512, maskable-512, apple-touch-icon-180, favicon.svg)
  are generated by `scripts/generate-icons.mjs` — a dependency-free PNG encoder,
  so no image tooling is required. `index.html` adds the iOS
  `apple-mobile-web-app-*` meta tags and apple-touch-icon for standalone use.
- Offline: the app shell and assets are precached; all data is local, so the app
  is fully usable offline.

## Accessibility

- Semantic controls (`<button>`, `<label>`+`<input>`, `<select>`), `aria-label`s
  on icon-only buttons, `role`/`aria-selected` on filter tabs, `aria-live` toast.
- Global `:focus-visible` outline; touch targets ≥ 44px (`--tap`).
- Inputs use ≥16px font to avoid iOS zoom-on-focus.
- `@media (prefers-reduced-motion: reduce)` disables transitions/animations.

## Deployment

- **`.github/workflows/ci.yml`** runs on push/PR: install → typecheck → lint →
  test → build.
- **`.github/workflows/deploy.yml`** runs on push to `main`: builds and publishes
  `dist/` to GitHub Pages via the official Pages actions.
- The Vite `base` is `/the-one-app/` (project page). If the repository is
  renamed, update `base` in `vite.config.ts` (and the manifest `start_url`/
  `scope` follow it automatically).

## Testing

- **Vitest** (jsdom, globals) with `fake-indexeddb` for storage tests.
- Coverage focuses on domain rules and persistence (see DATA_MODEL.md and the
  test files): the two-focus rule, task/action separation, backup validation,
  migrations, persistence round-trip, and today's-workout selection.
